以后要维护以下 **4 份核心文件夹**。

这些文档能将你的“宏观意图”转化为“执行细节”：

### 1. 核心状态拓扑图 (State Topology & Data Flow)
这是游戏的心脏。你需要一份文档定义 **“状态树”的完整结构**。
* **内容：** 详细列出 `publicStore`（公共池）、`playerStore`（玩家资产）和 `gameSystemStore`（回合逻辑）中每一个变量的类型和含义。
* **意义：** 这样同事在写新功能时，能立刻知道该去哪里取数据，而不是自己乱写局部状态，导致数据不同步。

### 2. 领域逻辑规约 (Domain Logic Specification - "The Rulebook")
要实现快速升级，必须将 **“游戏规则”与“UI组件”彻底解耦**。
* **内容：** 定义纯逻辑函数（Pure Functions）。例如 `calculatePurchaseCost(player, card)` 或 `validateTokenTake(tokens)`。
* **意义：** 如果以后规则变了（比如增加一种新宝石），你只需要改这一个逻辑文件，而不需要动任何 UI 代码。

### 3. 设计系统与组件规范 (Design System & Storybook)
针对你提到的美术改进需求。
* **内容：** 定义原子的 UI 规范（如：按钮的几种状态、边框的 BG3 风格纹理、不同层级卡牌的 CSS 类名）。
* **意义：** 保证以后你请了专业美术，他们可以直接替换 CSS 全局变量或图片资源，而不会破坏功能。

### 4. 版本升级与扩展指南 (Extensibility Guide)
专门为“快速升级”准备的手册。
* **内容：** 规定如何添加“扩展包”（如增加大师卡、增加特殊的贵族效果）。
* **设计原则：** 采用**插件式设计**或**策略模式**，让新功能通过“注入”而不是“修改源码”的方式实现。

### 关于“快速升级”的设计精要：
为了确保你提到的“快速升级”，请在文档中强调以下三点设计准则：

1.  **TypeScript 强类型协议：** 所有的卡牌、资源、玩家动作必须有严格的接口定义。这样当你修改一个规则时，编译器会自动告诉你哪些 UI 代码会崩掉，而不是等上线了才发现。
2.  **配置驱动（Config-Driven）：** 将卡牌数据、数值配置抽离到独立的 JSON 文件中。升级如果是单纯的数值调整，改 JSON 即可，无需重新编译代码。
3.  **解耦 UI 库：** 即使以后你想从 React 换到更先进的引擎，只要你的【逻辑层】（核心规则函数）是纯 JavaScript/TypeScript 写的，你的核心财产就永远可以无缝迁移。
